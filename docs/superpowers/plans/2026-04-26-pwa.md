# PWA Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make the SigMod Visualizer installable as a Progressive Web App on Android home screens, with offline-capable cache-first rendering.

**Architecture:** Two new static files (`manifest.json` and `sw.js`) at the site root, three small additions to `index.html` (manifest link, theme-color meta, inline registration script). No build step. Hand-rolled service worker — no Workbox or other tooling. User provides two icon PNGs.

**Tech Stack:** Web App Manifest (W3C), Service Worker API, vanilla JavaScript. No new dependencies.

**Spec:** [docs/superpowers/specs/2026-04-26-pwa-design.md](../specs/2026-04-26-pwa-design.md)

**Branch:** `pwa` (off `main`).

---

## How to verify

There is no test runner. Verification requires a local HTTPS-equivalent
server because service workers won't register on `file://`:

```bash
# in the project root
python -m http.server 8000
```

Then open `http://localhost:8000/` in Chrome/Edge. `localhost` counts as a
secure context, so the service worker will register.

DevTools paths used in verification:
- **Application → Manifest** — confirms `manifest.json` parsed, icons resolved.
- **Application → Service Workers** — confirms `sw.js` activated.
- **Network → Offline checkbox** — toggles the browser into offline mode.

## Task 2: Add the service worker and register it

**Files:**
- Create: `sw.js`
- Modify: `index.html` — append a registration `<script>` block at end of `<body>`

After this task, Chrome's address bar will show an "Install" icon (Mac/Windows desktop) or the install prompt will surface in the browser menu (Android Chrome). The service worker pre-caches the seven assets on install, then serves them from cache on subsequent visits.

### - [ ] Step 1: Create `sw.js` at the project root

```javascript
const CACHE_VERSION = 'sigmod-v1';
const PRECACHE = [
  './',
  './index.html',
  './style.css',
  './script.js',
  './manifest.json',
  './icon-192.png',
  './icon-512.png',
];

self.addEventListener('install', (event) => {
  event.waitUntil(
    caches.open(CACHE_VERSION).then((cache) => cache.addAll(PRECACHE))
  );
  self.skipWaiting();
});

self.addEventListener('activate', (event) => {
  event.waitUntil(
    caches.keys().then((keys) =>
      Promise.all(keys.filter((k) => k !== CACHE_VERSION).map((k) => caches.delete(k)))
    )
  );
  self.clients.claim();
});

self.addEventListener('fetch', (event) => {
  if (event.request.method !== 'GET') return;
  event.respondWith(
    caches.match(event.request).then((cached) => cached || fetch(event.request))
  );
});
```

The constants and functions:
- `CACHE_VERSION` — bump this string (e.g. `'sigmod-v2'`) whenever any of the seven assets change, so old caches get evicted on the next visit.
- `PRECACHE` — the seven URLs to cache up front. `./` is included separately from `./index.html` because some servers serve the index from the root URL only.
- `skipWaiting()` and `clients.claim()` make the new SW take control immediately rather than waiting for all tabs to close.
- The `fetch` handler skips non-GET requests (POST etc. wouldn't be cacheable anyway) and uses cache-first.

### - [ ] Step 2: Add the registration script to `index.html`

In `index.html`, find the existing `<script src="script.js"></script>` line near the end of `<body>` (just before `</body>`). Add a new `<script>` block immediately after it:

```html
  <script src="script.js"></script>
  <script>
    if ('serviceWorker' in navigator) {
      window.addEventListener('load', () => {
        navigator.serviceWorker.register('sw.js').catch(() => { /* SW unavailable; app still works online */ });
      });
    }
  </script>
</body>
```

The empty `catch` swallows registration errors silently. Service workers fail
on `file://` and on insecure HTTP (anything except `localhost` and HTTPS),
and we don't want to scare anyone reading the console with a noisy error in
those cases.

### - [ ] Step 3: Verify the service worker activates

In DevTools → Application → Service Workers (with `http://localhost:8000/` loaded):

- Status shows `activated and is running` next to `sw.js`
- "Source" link opens the file
- No errors in the small log at the top of the panel

If the SW shows as "redundant" or fails to activate, check the Console tab for an error from `sw.js`. The most common cause is a typo in `PRECACHE` (any URL that fails to fetch will abort the install). Hard-reload the page (`Ctrl+Shift+R`) after fixing.

### - [ ] Step 4: Verify offline mode

Still in DevTools, open Network tab. Tick the **Offline** checkbox at the top. Reload the page (`F5`).

Expected: the page renders fully, sliders work, all six modulation tabs render correctly. No network requests in the table show as failed in red — they all show "(ServiceWorker)" as the source.

If the page shows the browser's "no internet" dialog instead, the SW didn't precache something. Untick Offline, hard-reload to refresh the SW, then re-test.

### - [ ] Step 5: Verify installability

In Chrome's address bar, an "Install SigMod Visualizer" icon should appear at the right edge of the URL bar. Click it; Chrome offers to install. Click Install. The PWA opens in its own window with no browser chrome and the blue theme color in the title bar.

Uninstall after verification: open the installed PWA → ⋮ menu → "Uninstall SigMod Visualizer."

(On Android Chrome over LAN or HTTPS deploy, the equivalent is "Add to Home Screen" in the browser menu, producing a real home-screen icon.)

### - [ ] Step 6: Commit

```bash
git add sw.js index.html
git commit -m "feat: add service worker for offline-capable PWA"
```

## Task 3: Document the cache-version bump procedure in a code comment

**Files:**
- Modify: `sw.js` — add a one-line comment above `CACHE_VERSION`

The cache-version-bump rule is the single non-obvious operational thing about
this whole PWA. It's not in the spec because it's a code-level convention,
but it absolutely needs to live next to the constant or someone (probably
future-you) will edit `script.js`, reload, see the old version, and waste
half an hour debugging.

### - [ ] Step 1: Add the comment

In `sw.js`, replace the first line:

```javascript
const CACHE_VERSION = 'sigmod-v1';
```

with:

```javascript
// Bump this string whenever any precached asset changes, so old caches evict
// on the next visit. Otherwise users will see the stale cached version.
const CACHE_VERSION = 'sigmod-v1';
```

### - [ ] Step 2: Commit

```bash
git add sw.js
git commit -m "docs: explain CACHE_VERSION bump rule in sw.js"
```

## Task 4: Final manual verification

**Files:** none modified.

A short walkthrough that exercises everything across one fresh install +
offline cycle. Run after Tasks 1–3 are committed.

### - [ ] Step 1: Clean slate

In DevTools → Application → Storage, click **Clear site data**. This wipes the cache and unregisters the SW so the next reload behaves like a brand-new visitor.

### - [ ] Step 2: Fresh load, online

Reload `http://localhost:8000/`. Confirm:
- Page renders normally.
- DevTools → Application → Manifest shows the manifest parsed, both icons resolved.
- DevTools → Application → Service Workers shows `sw.js` activated.
- Network tab shows the second request batch (after the first reload) is served from the SW cache, not the network.

### - [ ] Step 3: Install, then go offline

Click the install icon in the address bar; install the PWA. The standalone window opens.

In that PWA window, open DevTools (`Ctrl+Shift+I`). Network → Offline. Reload.

Confirm: the page renders fully offline. All six modulation tabs work. Sliders respond.

### - [ ] Step 4: Update simulation

Untick Offline. Edit `style.css` — change any color value. Reload the PWA. The change does NOT appear (cache hit on the old version). Now bump `CACHE_VERSION` in `sw.js` from `'sigmod-v1'` to `'sigmod-v2'`. Reload twice (first reload registers the new SW; second reload serves from the new cache).

Confirm: the color change now appears.

After this verification, revert your `style.css` and `sw.js` test edits (or commit them as a real change if you wanted that color to ship).

### - [ ] Step 5: Uninstall the test install

Right-click the PWA window's title bar → "Uninstall SigMod Visualizer," or in Chrome's address bar of any tab open to the site, the install icon now shows as installed-and-clickable to uninstall.

This step is optional but keeps your machine tidy after testing.
